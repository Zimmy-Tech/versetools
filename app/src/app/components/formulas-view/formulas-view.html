Power System

Every ship has a fixed power budget generated by its power plant(s). Components draw from this budget via allocatable power pips. The power bar UI lets you redistribute pips within the total output.

Total Power Output

Single Power Plant:
  Total Power = Plant Output

Multiple Power Plants (pairing penalty):
  Each PP contributes = ceil( Output / 2 ) + Size

  Total Power = Σ( ceil( Outputi / 2 ) + Sizei )

A single power plant provides its full output as available pips. When two or more plants share a ship, each contributes roughly half its output plus a flat bonus equal to its size class (S1=+1, S2=+2, S3=+3).

Validated on Sabre (S1), Valkyrie (S2), and Redeemer (S3) with 6 configurations including mixed plants. Example: Sabre with SonicLite (13) + JS-300 (17) = ceil(13/2)+1 + ceil(17/2)+1 = 8 + 10 = 18 pips.

Total Power Used

Total Power Used = sum( Component Pips ) + Weapon Pips + Thruster Pips + Tool Pips + Tractor Pips

Cannot exceed Total Power Output. The UI clamps allocation changes to keep usage within budget.

Weapon Power Pool

Max Weapon Pips = min( Ship Pool Size, ceil( Total Weapon Power Cost ) )

where Total Weapon Power Cost = sum( Weapon Power Cost )  for all equipped weapons

The weapon pool is a fixed-size pool declared per-ship. Maximum usable pips are capped by whichever is smaller: the pool size or the total power cost of equipped weapons (rounded up).

Power Band Modifier

Band Modifier at Pips:
  if Pips <= 0  →  0  (component off)
  if no bands   →  1  (full efficiency)
  otherwise     →  efficiency of highest band whose threshold <= current Pips

Many components (shields, coolers, QDs) have tiered power bands. The modifier scales the component's effectiveness at the current pip level. Each band defines a pip threshold and the efficiency multiplier that applies above it.

Minimum Pip Block (Band-Gap)

Min Block = Band 2 Threshold - Band 1 Threshold    (if 2+ bands)
Min Block = 1                                       (if single band)

The minimum allocation for a component is derived from the gap between its first two power bands. This prevents the component from operating below its lowest effective threshold.

Default Power Allocation

Weapons 50% of pool size Shields 50% of max, at least min threshold Coolers minimum band pip Life Support 1 pip (always on) Radar max(1, round(Power Cost × Idle Consumption Fraction)) Quantum Drive 0 pips (off at spawn) Tools 1 pip per tool, togglable on/off (MOLE: 2 pips per turret) Tractor 2-pip block toggle

Weapons & DPS

Weapon damage calculations depend on weapon type (ballistic vs energy), fire rate, ammo mechanics, and heat management. DPS is computed over a 10-second window to capture burst/regen cycling.

Energy Weapon Ammo

Effective Magazine  = Base Magazine × Ship Ammo Multiplier
Magazine Cap        = floor( Effective Magazine )
Ammo at Pips(N)     = min( round( N × Effective Magazine / Total Weapon Power ), Magazine Cap )

where:
  Base Magazine        = weapon's default ammo capacity (typically 75 rounds)
  Ship Ammo Multiplier = per-ship engineering buff (ranges 1.75 – 6.0)
  Total Weapon Power   = sum of power cost across all weapons in the pool
  N                    = currently allocated weapon pips

Energy weapons share ammo from a pooled budget. Each weapon's share is proportional to its power cost relative to the total. More pips = more ammo per weapon.

Ballistic Weapon Burst

Burst Rounds = floor( Heat Capacity / Heat Per Shot )
Burst Time   = Burst Rounds / ( Cyclic Rate / 60 )

Ballistic weapons have fixed magazine capacity and fire until overheating. The burst window is determined by how fast heat accumulates versus the weapon's thermal ceiling.

Alpha Damage

Alpha Damage = Physical + Energy + Distortion + Thermal

Sum of all damage types dealt per shot. Each projectile carries a breakdown of damage across four categories.

Sustained DPS (10-second window)

Ballistic:
  Cycle Time  = Burst Time + Overheat Cooldown
  Full Cycles = floor( 10 / Cycle Time )
  Remaining   = 10 - Full Cycles × Cycle Time
  Total Dmg   = Full Cycles × Burst Dmg + min(Remaining, Burst Time) × Rounds/Sec × Alpha

Energy:
  Regen Time  = Regen Delay + Base Magazine / Regen Rate
  Cycle Time  = Burst Time + Regen Time
  (same cycle calculation as ballistic)

Computes total damage across burst/reload cycles within a 10-second window. This captures the real sustained output including downtime. "Regen Delay" is the pause before ammo begins regenerating; "Regen Rate" is ammo recovered per second.

Energy Weapon Sustained Ratio

Sustained Ratio(N) = (300 × N) / (300 × N + Cyclic Rate RPM)

where:
  300 = Regen Constant = (Magazine / Restock Cycles) × Regen Rate × 60
      = (75 / 3) × 15 = 300
  N   = weapon bars allocated (capped at max restock cycles, typically 3)

Fraction of theoretical max DPS achievable when sustaining fire continuously. The regen constant represents how quickly a weapon can replenish ammo between bursts. Higher ratio = less downtime.

Weapon Penetration

Penetration Distance = how far the projectile penetrates after impact (meters)
Penetration Radius   = min–max spread of the penetration cone (meters)

All weapons have penetration values. Distortion and scatter weapons
also have proximity detonation (separate min/max blast radius).

Higher penetration distance means the projectile passes through more material before stopping.

Missile Alpha & Blast Radius

Missile Alpha  = sum( Per-Missile Damage )  for all equipped missiles
Blast Radius   = Min Radius – Max Radius (meters) per missile

Total one-volley damage from all missile racks. Each missile also has an explosion radius defining the area of effect on detonation. Torpedoes have significantly larger blast radii than standard missiles.

Rate of Fire — Real DPS

This data is based on real in-game testing and results. It is my best understanding of how the system works, calculated to an error rate of less than 1.5%. It could possibly be incorrect, however, enough data points were checked that it seems incredibly likely.

The fire rates listed in the game data for many weapons are not what you actually get in-game. We tested this by timing ammo consumption across 9 different weapons and found a consistent pattern caused by server tick quantization.

Why Aren't Gatlings Affected?

There are two completely different firing systems in the game data:

Sequence weapons (repeaters & cannons) use discrete fire actions — each shot is an individual event. The server processes “fire barrel 1”, then on the next tick checks “time for barrel 2?”, and so on. Each shot must land on a server tick.

Gatlings use a continuous rapid-fire action. The weapon enters a “firing state” with a spin-up, then spawns projectiles using an accumulator. Instead of checking “is it time for the next shot?” each tick, it calculates how many shots should have fired since the last tick and spawns them all. A continuous timer tracks fractional shots between ticks — no per-shot alignment needed.

Cannons use the same sequence system as repeaters, but fire slowly enough (100–150 RPM) that the 33ms tick boundary never rounds them up.

The Formula

ticks = ceil(1800 / listed_RPM)
real_RPM = 1800 / ticks

Only applies to weapons using sequence fire actions with 2+ entries (repeaters). Gatlings and single-entry cannons are unaffected.

The Dead Zone

Any sequence weapon with a listed RPM of 601–899 gets rounded to 3 ticks = 600 RPM. An 825 RPM weapon fires at the same real rate as a 750 RPM weapon — but with lower per-shot damage because CIG balanced it for 825.

Shields

Shield mechanics use a primary/reserve model. The first two equipped shields are "active" (regen + HP + power pips). Additional shields beyond the second are "reserve" (HP pool only, no regen, auto-activate).

Total Shield HP

Total Shield HP = sum( Shield Health )  for ALL equipped shields (primary + reserve)

Shield Regen Rate (Linear Model)

Shield Regen Rate = Max Regen × ( Allocated Pips / Max Pips )

where:
  Max Regen      = sum( Shield Regen Speed )   for primary shields only (first 2)
  Allocated Pips = sum( Current Pips )         for primary shields only
  Max Pips       = sum( max(1, Power Ceiling - 1) )  for primary shields only

Regen scales linearly with allocated pips. Only the first two shields contribute to regeneration. Reserve shields (3rd+) add HP but never regen.

Shield Regen Time

Regen Time = Total Shield HP / Shield Regen Rate

Time to regenerate the full shield pool from zero. Practical time is longer due to damage-taken delays and downed-state recovery pauses.

Shield Power Distribution

Phase 1: Give each shield its minimum (band-gap) first
Phase 2: Distribute remaining pips up to each shield's max

Min Pips = Band 2 Threshold - Band 1 Threshold  (per shield)
Max Pips = Power Ceiling - 1                     (per shield)

The combined shield bar distributes pips across all primary shields, ensuring each meets its minimum threshold before allocating extras.

Resistances & Absorptions

Resistance scales with power pips:
  Resist at Pips = resistMin + (resistMax - resistMin) × (Pips / Max Pips)

  Positive = reduces damage taken by shield
  Negative = amplifies damage taken (vulnerability)
  Shield Damage = Incoming × (1 - Resist)

Absorption scales with power pips:
  Absorption at Pips = absMin + (absMax - absMin) × (Pips / Max Pips)

  1.0 = shield absorbs 100%, nothing reaches hull
  0.45 = 45% absorbed by shield, 55% bleeds through to hull
  Hull Bleedthrough = Incoming × (1 - Absorption)

Per damage type: Physical, Energy, Distortion

Both resistance and absorption scale linearly with allocated shield pips, from minimum values at 1 pip to maximum at full power. Physical damage has low resist (0–25%) and low absorption (0–45%), meaning ballistic weapons damage both shield and hull simultaneously. Energy damage is amplified (-100% to -10% resist) but fully absorbed (100%), dealing more shield damage but none to hull.

Signatures (EM / IR / CS)

Ship signatures determine detectability by sensors and missile locks. Three types: Electromagnetic (EM), Infrared (IR), and Cross-Section (CS). EM and IR formulas are validated against in-game values. CS research is ongoing.

Electromagnetic (EM) Signature

EM = sum( Component EM )

  Power Plants:  emMax × ( Total Power Used / Total Power Output )
  Weapons:       emSignature   (if weapon pool > 0, else 0)
  Others:        emMax × Band Modifier at Current Pips
    (Shields, Coolers, QD, Life Support, Radar)

Power plant EM scales with power utilization — freeing pips reduces plant signature. Validated via Aurora SE cooler-swap test (predicted delta 2,230 vs game delta 2,200). The ship armor signalEM multiplier does NOT apply to the displayed total.

Infrared (IR) Signature

IR = irMax × max( MCF, Demand / Supply ) × Armor signalIR

where:
  irMax    = sum( Cooler irSignature × Band Modifier )  for active coolers
  MCF      = weighted average of active coolers' Min Consumption Fraction
  Demand   = total cooling demand (all components + thrusters)
  Supply   = total cooling supply (all coolers at current pips)

At normal load, coolers idle at their MCF fraction (e.g., 0.333 for Bracer coolers). When overloaded (one cooler off), the demand/supply ratio takes over and IR spikes. Unlike EM, the armor IR multiplier IS applied to the displayed value. Thrusters contribute to cooling demand (~1 unit per pip) despite having 0 static IR.

Cross-Section (CS)

CS = Base Hull CS × Armor CS Multiplier

Armor CS Multiplier:
  Standard ships: 1.0 – 1.5 (higher = more visible)
  Stealth ships:  0.55 – 0.85 (Eclipse 0.6, Prowler 0.55, Ghost 0.72)
  Heavy ships:    1.13 – 1.4 (Hornets 1.13, Vanguard 1.33, Starfarer 1.4)

Estimation formula (approximate):
  CS ≈ 253 × mass^0.33 × CS Multiplier^1.4

CS is primarily driven by the ship's physical hull geometry (a per-ship base value), multiplied by the armor's CS modifier. The base CS cannot be reliably calculated from mass or dimensions alone — ships with similar mass can have very different CS values (e.g., Talon 3k vs Eclipse 4.3k at nearly identical mass and CS multiplier).

Validated ships show the exact in-game value. All others display an estimate based on a mass-based approximation (marked "EST"). The estimate is within ~10% for fighters but can deviate up to 25% for larger ships.

12 ships validated in-game so far: Arrow (7.4k), Gladius (9.7k), Hawk (11.2k), Eclipse (4.3k), Talon (3.0k), 400i (19.7k), Vanguard (22.7k), Andromeda (23.9k), Spirit C1 (24.7k), 600i (24.7k), Starlancer TAC (18.2k), Polaris (42.4k).

Stealth Components

TYDT Stealth coolers (lowest IR/EM) and stealth power plants RACO Stealth quantum drive (Drift: 7,500 vs 18,000 standard S1) ASAS Stealth shields (Obscura/Umbra: 500 EM vs 750+ standard)

Armor

Ship armor provides HP, damage deflection, and damage/durability modifiers. Armor is a fixed property of each ship hull — not swappable.

Armor Deflection

If Weapon Alpha <= Armor Deflect Value  →  Damage = 0  (deflected)
If Weapon Alpha >  Armor Deflect Value  →  Full damage passes through

Separate deflect thresholds for Physical and Energy damage.

Weapons below the deflect threshold deal zero damage. The Armor Check panel shows which weapons penetrate and which are stopped, with the 6 closest weapons to the threshold highlighted.

Armor Damage per Shot — CONFIRMED

Armor Damage = Weapon Alpha × Damage Modifier

  Modifier < 1.0  →  armor takes reduced damage (e.g., 0.5 = half damage)
  Modifier = 1.0  →  full damage
  Modifier > 1.0  →  armor takes amplified damage

Per-type: Physical, Energy, Distortion

Shots to Strip Armor = ceil( Armor HP / (Weapon Alpha × Damage Modifier) )

Validated in-game across three ships with different modifiers:

Asgard 0.50 energy → 11.6%/shot predicted, ~11.5% observed Guardian MX 1.10 energy → 42.0%/shot predicted, 42% observed Polaris 0.60 energy → 5.0%/shot predicted, 5.0% observed

Hull Bleedthrough

While armor is up, damage simultaneously bleeds through to hull.
Bleedthrough rate increases as armor degrades.

Ion vs Asgard (no shields):
  Shot 1 (armor 100%) → 3% hull lost
  Shot 5 (armor  42%) → 4–5% hull lost
  Shot 9 (armor   0%) → 7% hull lost

The exact bleedthrough formula is under investigation. Damage is not purely sequential (armor first, then hull) — hull takes increasing damage as armor thins out. Capital ships may behave differently (Polaris showed zero hull bleedthrough through 20 shots).

Durability Modifiers — Under Investigation

Per-type: Physical, Energy

Role unconfirmed. Does NOT control per-shot armor damage (disproved by testing).
May affect hull bleedthrough scaling or component penetration.

Previously assumed to control armor HP degradation rate, but in-game testing shows armor damage is governed by the Damage Modifier field instead. Further testing is needed to determine what Durability actually affects.

Penetration Multipliers

Fuse Penetration   = chance of projectile reaching internal fuses
Component Penetration = chance of hitting internal components

Values are multipliers (0.0 = fully protected, 1.0 = no protection)

These determine how vulnerable a ship's internal systems are to damage that passes through the armor. Lower values = better protection.

Cooling

Coolers generate coolant that other components consume. If cooling demand exceeds supply, components overheat. The cooling gauge shows the supply/demand ratio as a percentage.

Cooling Supply (per cooler)

Cooler Supply = coolingRate × pips × bandMod(pips) / maxPips

maxPips    = powerMax - 1
bandMod    = efficiency tier at current pip level
if Pips = 0  →  Supply = 0

Each cooler's output is the product of its base cooling rate, the fraction of power it's receiving (pips/maxPips), and its efficiency at that power level (band modifier). Total supply is the sum across all equipped coolers.

At max pips, pips/maxPips = 1 and bandMod = 1.0, so supply equals the base cooling rate. Below max, output drops from both reduced power AND reduced efficiency.

Cooling Demand

Total Demand = PP_IDLE + Σ( pips × weight )

Two-tier weight system (heat generated per pip):
  High tier (~2.0/pip):
    Life Support     2.300       Quantum Drive    2.070
    Radar            1.988       Shields          1.978

  Low tier (~1.0/pip):
    Thrusters        1.032       Tools/Tractors   0.966
    Coolers          0.939       Weapons          0.900

PP_IDLE = 0.04

Components fall into two tiers: power-hungry systems (Radar, Shields, QD, Life Support) generate ~2 units of heat per pip, while lighter systems (Weapons, Coolers, Thrusters, Tools) generate ~1 unit per pip.

PP idle base is essentially zero (0.04). Validated across Aurora MK II, Guardian, Guardian MX, Crusader Intrepid, and Asgard with isolated component tests. Max error: 2% within game’s whole-percentage rounding. Per-ship overrides exist for Polaris and Mercury Star Runner.

Cooling Percentage

Cooling % = round( Total Demand / Total Supply × 100 )

Displayed on the engineering cooling gauge. Values above 100% indicate thermal overload. When cooling supply drops below demand, the reading escalates rapidly (observed 152% → 457% on Apollo Medivac with one cooler at minimum).

Research Notes

A cooling multiplier of 2.5 exists in the game's resource network configuration. Some tools use this as a demand multiplier, producing much higher cooling percentages. In-game testing shows stable IR and normal operation at configs where the 2.5× multiplier would predict overload, disproving its use in the displayed cooling calculation.

The corrected supply formula (rate × pips × bandMod / maxPips) was discovered through cooler toggle tests — at reduced pips, the game applies BOTH the band efficiency penalty AND a linear power fraction, not just one.

Weapon demand was initially modeled as scaling with powerDraw (0.484 × totalPD), but isolated high-pip tests on the Guardian (4w, cooler only = 16%) disproved this. The powerDraw formula appeared to work because early tests used low weapon pips in multi-component configs where the weapon term was too small to distinguish from a fixed weight. The corrected fixed weight of 0.90/pip was validated across 5 ships with max 2% error.

Cooling System Theory

The in-game cooling gauge reflects real-time thermal state, not a static calculation. Below 100%, it approximates steady-state equilibrium. Above 100%, the system enters thermal runaway where insufficient cooling causes heat accumulation that further overwhelms the system.

The gauge is essentially a thermometer. Our formulas approximate the steady-state equilibrium point that the gauge settles at during normal operation.

Flight & Thrusters

Flight stats are both crowd sourced and calculated. Rotation rates scale with thruster power pips between base and boosted values.

Rotation Rate Scaling

Thruster Multiplier = (Current Thruster Pips - 1) / (Max Thruster Pips - 1)
  (if Max Thruster Pips <= 1: Thruster Multiplier = 1)

Rotation Rate = Base Rate + Thruster Multiplier × (Boosted Rate - Base Rate)

Rotation rates (pitch, yaw, roll) interpolate linearly between their base value (at 1 pip) and their boosted value (at max pips). The thruster power bar controls this scaling.

Speed Stats

SCM Speed         —  Standard Combat Maneuvering speed
NAV Speed         —  Navigation (cruise) speed
Boost Speed (Fwd) —  Afterburner forward speed
Boost Speed (Bwd) —  Afterburner reverse speed

These values come from the ship's flight controller configuration.

Acceleration

Directional: Forward, Retro, Strafe, Up, Down
Afterburner: AB Forward, AB Retro, AB Strafe, AB Up, AB Down

Each ship displays:
  Tested    = date the values were last verified in-game
  Checked by = who verified or approved the data

Acceleration values are community-tested and submitted via the Submit Data tab. Submissions are reviewed before being published. Ships with missing or stale data (older than 90 days) are flagged for re-testing. The "Help Us" button in the loadout view links directly to the submission form.

Mining

Mining lasers have base stats that are modified by equipped mining modules. Laser modifiers are multiplicative; module modifiers are additive.

Mining Power with Modules

Combined Power Mult = product( Module Power Multiplier )  for all equipped modules

Combined Min Power = round( Laser Min Power × Combined Power Mult )
Combined Max Power = round( Laser Max Power × Combined Power Mult )

Power multipliers from modules stack multiplicatively. Min and max laser power are both scaled by the combined multiplier.

Mining Stat Modifiers

For each stat (Instability, Optimal Window, Optimal Rate,
                Resistance, Shatter Damage, Inert Materials, Overcharge):

  Total Stat = Laser Base Value + sum( Module Adjustment )  for all equipped modules

Module stats stack additively onto the laser's base values. Values are percentage adjustments (e.g., -35 = -35% instability reduction).

Mineral Spawn Locations

Each location has three mining categories:
  Ship Mining    —  asteroid deposits (Prospector, MOLE)
  ROC Mining     —  surface deposits (ground vehicles)
  Hand Mining    —  harvestable gems (on foot)

Each category lists minerals with:
  Mineral   = ore or gem type
  Tier      = Common, Uncommon, Rare, Epic, Legendary
  Chance %  = relative probability of spawning at that location

47 locations across Stanton, Pyro, and Nyx systems are mapped with full mineral distributions. The Mining tab lets you drill from system to planetary body to see what spawns where.

Mineral Properties

Each mineral has intrinsic properties that affect mining difficulty:
  Instability         = how volatile the rock is while mining
  Resistance          = how hard the rock is to crack
  Optimal Window      = size of the safe power zone
  Window Variance     = randomness in the optimal window
  Optimal Thinness    = how narrow the sweet spot is
  Explosion Multiplier = damage if the rock detonates
  Cluster Factor      = likelihood of finding grouped deposits

These properties determine how carefully you need to manage laser power when mining each mineral type. High instability + high resistance minerals (like Quantainium) require precise power control.

Radar

Radar lock range scales linearly with power allocation between minimum and maximum aim distances.

Lock Range

Pip Fraction = min( Current Pips / Max Pips, 1 )
Lock Range   = round( Min Lock Distance + (Max Lock Distance - Min Lock Distance) × Pip Fraction )

where Max Pips = radar's total power cost

At minimum power, lock range equals the radar's shortest lock distance. At full power, it reaches maximum lock distance. The scaling is linear between these bounds.

Radar Minimum Power

Radar Min Pips = max( 1, round( Power Cost × Idle Consumption Fraction ) )

The minimum allocation is derived from the radar's idle consumption fraction (typically 0.25). This is the lowest power level at which the radar operates.

Cargo

Cargo capacity is calculated from the interior dimensions of each cargo grid in the ship's default loadout.

SCU Capacity

SCU Per Grid = floor(Width / 1.25) × floor(Length / 1.25) × floor(Height / 1.25)
Total Cargo  = sum( SCU Per Grid )  for all cargo grids in default loadout

Each axis is divided by the SCU unit size (1.25m) and floored. The product gives the number of 1-SCU boxes that fit in that grid.

Crafting

The crafting system uses quality-based modifiers on ingredients to determine the final stats of crafted items. Blueprints are unlocked through mission rewards.

Quality Modifier Interpolation

t        = (Clamped Quality - Range Start) / (Range End - Range Start)
Modifier = Start Multiplier + t × (End Multiplier - Start Multiplier)

Clamped Quality = clamp( Quality, Range Start, Range End )
Quality range: 0 – 1000

Each ingredient has quality modifiers that map a quality value (0–1000) to a stat multiplier via linear interpolation between the curve's start and end points.

Combined Modifier (Multi-Ingredient)

Combined Modifier = product( Ingredient Modifier )  per stat

Each ingredient's modifier for a given stat multiplies together.

When multiple ingredients affect the same stat, their modifiers are multiplicative. A modifier of 1.0 is neutral, >1.0 is a buff, <1.0 is a nerf.

Dismantle Returns

Returned Quantity = floor( Original Quantity × 0.5 )

Resources < 0.01 SCU return nothing.
